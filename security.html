WEB SECURITY

웹 해킹 기법과 공격 원리

웹 보안은 공격 도구를 외우는 일이 아니라, 사용자의 입력이 서버와 브라우저에서 어떻게 처리되는지 추적하는 일에서 시작됩니다.

XSS브라우저

Cross-Site Scripting

검증되지 않은 입력이 HTML로 출력될 때 악성 스크립트가 실행되는 취약점입니다.

공격 원리

댓글, 검색어, 프로필 이름 같은 입력값이 HTML로 해석되면 공격자가 작성한 script가 다른 사용자의 브라우저에서 실행될 수 있습니다. 

// 위험
result.innerHTML = userInput;

// 방어
result.textContent = userInput;

대응: 출력 인코딩, HTML sanitize, Content Security Policy, 쿠키 HttpOnly 설정.

SQLi데이터베이스

SQL Injection

사용자 입력이 SQL 문장에 직접 합쳐질 때 쿼리 구조가 변조되는 공격입니다.

공격 원리

로그인 폼의 아이디나 검색어가 SQL 문자열에 그대로 연결되면 조건식이 변조되어 인증 우회나 데이터 유출이 발생할 수 있습니다. 

// 위험
sql = "SELECT * FROM users WHERE id = '" + id + "'";

// 방어
sql = "SELECT * FROM users WHERE id = ?";
db.query(sql, [id]);

대응: Prepared Statement, 입력값 형식 검증, DB 계정 최소 권한.

CSRF요청 위조

Cross-Site Request Forgery

로그인된 사용자의 권한을 이용해 원치 않는 요청을 보내게 만드는 공격입니다.

공격 원리

사용자가 로그인한 상태에서 악성 페이지를 방문하면, 브라우저가 쿠키를 자동으로 포함해 상태 변경 요청을 보낼 수 있습니다. 

<form action="https://site.example/delete" method="POST">
  <input name="postId" value="10">
</form>

대응: CSRF 토큰, SameSite 쿠키, 중요 요청 재인증.

Access권한

Broken Access Control

URL이나 파라미터만 바꿔 다른 사용자의 데이터에 접근할 수 있는 문제입니다.

공격 원리

서버가 “로그인했는가”만 확인하고 “이 데이터의 소유자인가”를 확인하지 않으면 다른 사용자의 리소스를 볼 수 있습니다. 

// 위험한 패턴
GET /orders/1024

// 서버에서 반드시 확인
if (order.userId !== loginUser.id) {
  return 403;
}

대응: 객체 단위 권한 확인, 관리자 기능 서버 검증, 기본 거부 정책.

Auth인증

인증 실패

약한 비밀번호 정책, 세션 만료 미흡, 로그인 시도 제한 부재는 계정 탈취 위험을 높입니다.

공격 원리

무차별 대입, 세션 고정, 예측 가능한 토큰은 공격자가 계정 접근 권한을 얻는 통로가 됩니다. 

Set-Cookie: session=...; HttpOnly; Secure; SameSite=Lax

대응: 강한 비밀번호 저장 해시, MFA, 로그인 제한, 안전한 쿠키 옵션.

Upload파일

위험한 파일 업로드

확장자와 MIME 타입, 저장 경로, 실행 권한을 검증하지 않으면 서버에서 악성 파일이 실행될 수 있습니다.

공격 원리

공격자가 스크립트 파일을 이미지처럼 업로드하고 서버가 이를 실행 가능한 위치에 저장하면 원격 코드 실행으로 이어질 수 있습니다. 

// 점검 항목
허용 확장자만 받기
파일 크기 제한
업로드 경로 실행 권한 제거
파일명 랜덤화

대응: 허용 목록 검증, 저장소 분리, 바이러스 검사, 다운로드 시 Content-Disposition 사용.

참고 기준: OWASP Top 10:2021, OWASP A03 Injection

공격 흐름을 읽는 방법

취약점은 대부분 입력, 처리, 출력, 권한 확인 중 하나가 빠졌을 때 발생합니다.

방어 전략 보기 OWASP Top 10
INPUT

신뢰할 수 없는 입력 수집

폼, URL 파라미터, 쿠키, 헤더, 파일은 모두 조작될 수 있는 데이터로 가정합니다.

LOGIC

검증 없이 비즈니스 로직 실행

권한 확인이나 형식 검사가 빠지면 의도하지 않은 데이터 조회와 변경이 가능합니다.

OUTPUT

출력 단계에서 코드로 해석

HTML, JavaScript, SQL, Shell처럼 실행 의미가 있는 문맥에서는 반드시 이스케이프가 필요합니다.
